# Data Quality Rules

## Step Order in SSIS Data Flows

For all fact flows, the order is:
1. **Deduplication (Sort)** — remove duplicate country + date rows from source
2. **Data Conversion** — cast all columns to target types; cast failures → `dq_rejected_rows` (DQ-CAST)
3. **DQ Filter (Conditional Split)** — validate typed values; failures → `dq_rejected_rows`
4. **Lookups** — resolve surrogate keys; no-match → `dq_rejected_rows`

Data Conversion runs before the DQ Filter so that all filter expressions (`date > GETDATE()`, `new_cases < 0`) operate on correctly typed values, not raw strings.

## Validation Rules (applied in SSIS before load)

| # | Rule | Column(s) | SSIS Step | Action if Violated |
|---|---|---|---|---|
| DQ-CAST | Source value cannot be cast to target type (e.g., `"N/A"` in a numeric column, malformed date string) | All columns in Data Conversion | Data Conversion error output | Redirect row → `dq_rejected_rows` |
| DQ-01 | `date` must not be null | `date` (DATE) | Conditional Split | Redirect row → `dq_rejected_rows` |
| DQ-02 | `date` must not be in the future (`date > GETDATE()`) | `date` (DATE) | Conditional Split | Redirect row → `dq_rejected_rows` |
| DQ-03 | `continent` must not be null | `continent` | Conditional Split | Redirect row — removes all aggregate/non-country rows automatically |
| DQ-04 | `new_cases` must not be negative | `new_cases` (FLOAT) | Conditional Split | Redirect row → `dq_rejected_rows` |
| DQ-05 | `new_deaths` must not be negative | `new_deaths` (FLOAT) | Conditional Split | Redirect row → `dq_rejected_rows` |
| DQ-06 | `total_cases` must never decrease day over day | `total_cases` | Post-load check | Flag as anomaly, log warning in `etl_verification_log` |
| DQ-07 | `total_deaths` must never decrease day over day | `total_deaths` | Post-load check | Flag as anomaly, log warning in `etl_verification_log` |
| DQ-08 | `total_vaccinations` must never decrease day over day | `total_vaccinations` | Post-load check | Flag as anomaly, log warning in `etl_verification_log` |
| DQ-09 | `positive_rate > 1` — positivity rate above 100% is impossible | `positive_rate` (FLOAT) | Conditional Split (fact_covid_cases) | Redirect row → `dq_rejected_rows` |
| DQ-10 | `stringency_index > 100` — Oxford index is defined 0–100 | `stringency_index` (FLOAT) | Conditional Split (fact_covid_cases) | Redirect row → `dq_rejected_rows` |

## Reject Table

Rows that fail any DQ rule or lookup are written to `dq_rejected_rows` with:
- Original row data (raw country, date, new_cases, new_deaths)
- Rule ID that was violated (`DQ-CAST`, `DQ-01` to `DQ-05`, `LOOKUP-LOC`, `LOOKUP-DATE`)
- Source file name
- Load timestamp

## Null Policy

### Governing Principle

**NULL means "not reported by the source" — it is never substituted with zero.**

Replacing NULL with 0 would be incorrect for COVID data: a NULL in `new_cases` means the country did not report that day, not that zero cases occurred. Downstream analytical queries must use `ISNULL` / `COALESCE` where needed rather than relying on the warehouse to pre-fill zeros.

The only columns that are guaranteed non-null are keys and columns enforced by DQ rules. All metric columns carry nulls through as-is.

### Analytical Query Guidance for Nulls

Queries that aggregate nullable metric columns must account for nulls explicitly:

```sql
-- Correct: AVG ignores nulls by default in SQL Server
SELECT AVG(new_cases) FROM fact_covid_cases WHERE location_id = 1;

-- Correct: SUM of nulls = null — use ISNULL when a zero total is needed
SELECT ISNULL(SUM(new_cases), 0) FROM fact_covid_cases WHERE location_id = 1;

-- Correct: exclude countries with no hospitalization data before comparing
SELECT location_id, AVG(daily_occupancy_hosp)
FROM fact_hospitalization
WHERE daily_occupancy_hosp IS NOT NULL
GROUP BY location_id;
```

## Outlier Detection

### Two-Layer Approach

| Layer | Type | Where enforced | Action |
|---|---|---|---|
| Hard boundaries | Physically impossible values — always wrong regardless of context | SSIS Conditional Split (DQ-09, DQ-10) | Reject row → `dq_rejected_rows` |
| Soft outliers | Statistically unusual but potentially valid — OWID sometimes dumps accumulated historical corrections as a single large spike | Post-load CHECK 11 in `usp_verify_etl_load` | Flag in `etl_verification_log` — does not fail the package |

### Hard Boundary Rules (enforced in SSIS — row rejected)

Added to the Conditional Split in the relevant fact flow. These run after Data Conversion so comparisons operate on typed FLOAT values.

| # | Rule | Column | Flow | Rationale |
|---|---|---|---|---|
| DQ-09 | `positive_rate > 1` | `positive_rate` | fact_covid_cases | Positivity rate is a proportion (0–1). Values above 1 (>100%) are impossible. |
| DQ-10 | `stringency_index > 100` | `stringency_index` | fact_covid_cases | Oxford stringency index is defined on a 0–100 scale. Values above 100 are impossible. |

> **Why not reject `new_cases > population`?** This check requires joining to `dim_location` for the country's population, which is not available at the Conditional Split stage. It is instead flagged as a soft outlier in the post-load check.

### Soft Outlier Rules (post-load check — flagged, not rejected)

OWID sometimes publishes accumulated backlog corrections as a single-day spike. A spike of 500k new cases in one day for a large country may be valid. These are flagged for human review but do not fail the package.

| # | Rule | Column | Threshold | Rationale |
|---|---|---|---|---|
| OL-01 | Single-day case spike | `new_cases` | > 1,000,000 | Plausible only for the largest countries during extreme backlog corrections; always worth reviewing |
| OL-02 | Reproduction rate extreme | `reproduction_rate` | > 15 | Reproduction rates above 15 have no epidemiological precedent |
| OL-03 | Logical inconsistency in vaccination | `people_vaccinated`, `people_fully_vaccinated` | `people_vaccinated < people_fully_vaccinated` | Fully vaccinated cannot exceed total vaccinated — signals a data integrity issue |
| OL-04 | Case count exceeds population | `new_cases`, `population` | `new_cases > population` (joined via dim_location) | Impossible for a single day — a country cannot have more new cases than its entire population |

All soft outlier results are written to `etl_verification_log` (CHECK 11) as informational — the package continues to completion.

## Row Count Logging (In vs Out)

### Purpose

Track how many rows were extracted from each source, how many were loaded, and how many were rejected — per run, per Data Flow. This makes it possible to spot silent data loss (rows that neither loaded nor rejected) and compare run sizes over time.

### SSIS Implementation

Add the following components to each fact Data Flow:

| Component | Type | Placement | SSIS Variable |
|---|---|---|---|
| Row Count — Extracted | Row Count | Immediately after Flat File Source, before Sort | `@[User::RowsExtracted]` |
| Row Count — Loaded | Row Count | On the good-row path, just before OLE DB Destination | `@[User::RowsLoaded]` |
| Row Count — Rejected | Row Count | On the rejected-row path, just before dq_rejected_rows Destination | `@[User::RowsRejected]` |

After each Data Flow Task, wire an Execute SQL Task in the Control Flow to write the counts to `etl_run_log`:

```sql
INSERT INTO dbo.etl_run_log
    (flow_name, source_file, rows_extracted, rows_loaded, rows_rejected, rows_discarded)
VALUES
    (?, ?, ?, ?, ?, ? - ? - ?)
-- Parameters (mapped to SSIS variables in order):
-- 1. @[User::FlowName]       e.g. 'fact_covid_cases'
-- 2. @[User::SourceFile]     e.g. 'owid_covid_compact.csv'
-- 3. @[User::RowsExtracted]
-- 4. @[User::RowsLoaded]
-- 5. @[User::RowsRejected]
-- 6. @[User::RowsExtracted] - @[User::RowsLoaded] - @[User::RowsRejected]  (discarded)
```

### Querying Run History

```sql
-- Compare last 5 runs for fact_covid_cases
SELECT TOP 5
    run_timestamp,
    rows_extracted,
    rows_loaded,
    rows_rejected,
    rows_discarded,
    FORMAT(CAST(rows_rejected AS FLOAT) / NULLIF(rows_extracted, 0) * 100, 'N1') AS reject_pct
FROM dbo.etl_run_log
WHERE flow_name = 'fact_covid_cases'
ORDER BY run_timestamp DESC;
```

### Silent Loss Check

`rows_discarded = rows_extracted - rows_loaded - rows_rejected` must always be 0. A non-zero value means rows entered the pipeline but were neither loaded nor rejected — this indicates a misconfigured SSIS error output and must be investigated immediately. CHECK 2 in `usp_verify_etl_load` catches null FK rows, but `rows_discarded` in the run log catches the problem earlier.

## Rejection Threshold Policy

### Purpose

Without a threshold, the SSIS package succeeds even if 99% of rows are rejected — silently producing an empty or near-empty warehouse. A threshold check fires a hard error and fails the package if unexpected rejections exceed an acceptable level.

### Expected vs Unexpected Rejects

Not all rejects are equal. DQ-03 (continent IS NULL) removes OWID aggregate rows such as "World", "Africa", "High-income countries" — these are **expected** and excluded from the threshold calculation. Only the rules below count toward the threshold:

| Rule | Type | Counts toward threshold? |
|---|---|---|
| DQ-CAST | Unexpected | Yes |
| DQ-01 | Unexpected | Yes |
| DQ-02 | Unexpected | Yes |
| DQ-03 | Expected (aggregate row removal) | **No** |
| DQ-04 | Unexpected | Yes |
| DQ-05 | Unexpected | Yes |
| LOOKUP-LOC | Unexpected | Yes |
| LOOKUP-DATE | Unexpected | Yes |

### Thresholds per Source

| Source File | Fact Table | Threshold | Rationale |
|---|---|---|---|
| `owid_covid_compact.csv` | `fact_covid_cases` | > 5% unexpected rejects | Reliable source; ISO-3 join; high threshold would mask real data issues |
| `vaccinations_global.csv` | `fact_vaccination` | > 10% unexpected rejects | Name-based country join is riskier; some countries may have spelling mismatches |
| `hospital.csv` | `fact_hospitalization` | > 5% unexpected rejects | ISO-3 join is reliable; small file so even a few failures are significant |

### Implementation in usp_verify_etl_load

Add a threshold check as part of the post-load verification stored procedure. The check computes unexpected reject % for each source file relative to rows loaded into the corresponding fact table.

```sql
-- Threshold check — added to usp_verify_etl_load
DECLARE @run_start DATETIME2 = (SELECT MIN(load_timestamp) FROM dbo.dq_rejected_rows
                                 WHERE load_timestamp >= CAST(GETDATE() AS DATE));

-- fact_covid_cases threshold
DECLARE @loaded_cases  INT = (SELECT COUNT(*) FROM dbo.fact_covid_cases);
DECLARE @rejected_cases INT = (SELECT COUNT(*) FROM dbo.dq_rejected_rows
                                WHERE source_file = 'owid_covid_compact.csv'
                                  AND rule_id <> 'DQ-03'        -- exclude expected aggregate removes
                                  AND load_timestamp >= @run_start);
IF CAST(@rejected_cases AS FLOAT) / NULLIF(@loaded_cases + @rejected_cases, 0) > 0.05
    RAISERROR('Reject threshold exceeded for fact_covid_cases: %d unexpected rejects out of %d total rows.', 16, 1, @rejected_cases, @loaded_cases + @rejected_cases);

-- fact_vaccination threshold
DECLARE @loaded_vacc   INT = (SELECT COUNT(*) FROM dbo.fact_vaccination);
DECLARE @rejected_vacc  INT = (SELECT COUNT(*) FROM dbo.dq_rejected_rows
                                WHERE source_file = 'vaccinations_global.csv'
                                  AND load_timestamp >= @run_start);
IF CAST(@rejected_vacc AS FLOAT) / NULLIF(@loaded_vacc + @rejected_vacc, 0) > 0.10
    RAISERROR('Reject threshold exceeded for fact_vaccination: %d unexpected rejects out of %d total rows.', 16, 1, @rejected_vacc, @loaded_vacc + @rejected_vacc);

-- fact_hospitalization threshold
DECLARE @loaded_hosp   INT = (SELECT COUNT(*) FROM dbo.fact_hospitalization);
DECLARE @rejected_hosp  INT = (SELECT COUNT(*) FROM dbo.dq_rejected_rows
                                WHERE source_file = 'hospital.csv'
                                  AND load_timestamp >= @run_start);
IF CAST(@rejected_hosp AS FLOAT) / NULLIF(@loaded_hosp + @rejected_hosp, 0) > 0.05
    RAISERROR('Reject threshold exceeded for fact_hospitalization: %d unexpected rejects out of %d total rows.', 16, 1, @rejected_hosp, @loaded_hosp + @rejected_hosp);
```

### Behaviour on Breach

- `RAISERROR` with severity 16 causes the Execute SQL Task in SSIS to fail
- SSIS marks the package as Failed and stops execution
- The `etl_verification_log` table records the failure with the check name and reject count
- The warehouse is left in a truncated state (fact tables empty or partially loaded) — do not use for reporting until the next successful run

## Late Arriving Data Strategy

### Governing Principle

**The full reload strategy makes late arriving data a non-issue for fact tables.** Because every package run truncates and reloads all three fact tables from the current CSV snapshot, historical corrections made by OWID are automatically picked up on the next run — no special logic is needed.

### Scenarios

| Scenario | What Happens | Handled? |
|---|---|---|
| OWID corrects a historical case count (e.g., USA 2021-03-15 changes from 50,000 to 48,000) | Next full reload picks up the corrected value — old value is replaced by the truncate | ✅ Automatic |
| OWID adds retrospective data for a country that previously had no records on that date | Next full reload includes the new rows | ✅ Automatic |
| A new country appears in the source CSV | dim_location upsert inserts the new country; fact rows load on the same run | ✅ Automatic |
| OWID corrects a country's population or GDP in the compact CSV | dim_location uses IF NOT EXISTS upsert — the existing row is **not** updated | ❌ Not handled — see note below |
| A record arrives with a date before 2020-01-01 | dim_date starts at 2020-01-01; the date_id lookup will fail → row goes to `dq_rejected_rows` (LOOKUP-DATE) | ⚠️ Rejected — acceptable, OWID data does not pre-date 2020 |
| A record arrives with a date after today (future date) | Caught by DQ-02 before lookup — rejected to `dq_rejected_rows` | ✅ Handled by DQ-02 |

### dim_location Late Arriving Updates

dim_location uses SCD Type 1 with an IF NOT EXISTS upsert — meaning the first loaded value for each country is kept. If OWID later corrects a country's `population`, `gdp_per_capita`, or other demographic, the existing row in dim_location will **not** be updated automatically.

**Workaround if a dim_location update is needed:**
1. Run the following in SSMS to clear the affected country:
   ```sql
   DELETE FROM dbo.dim_location WHERE code = 'GBR';  -- replace with target ISO-3 code
   ```
2. Re-run the SSIS package — the dim_location flow will re-insert the country with current values from the CSV.

This is an accepted trade-off for a learning project. Country metadata changes infrequently and the manual workaround is straightforward.

### Run Cadence

OWID publishes dataset updates daily. Running the SSIS package once per day (via SQL Server Agent) ensures the warehouse stays current and all historical corrections are picked up within 24 hours of OWID publishing them.

## Computed Fields — Two Origins

Fields in the warehouse come from two distinct origins. Understanding which is which prevents incorrect attempts to re-derive OWID fields in SSIS.

### 1. Pre-computed by OWID — ETL passes through as-is

These columns already exist in the source CSV. SSIS does **not** compute them — it casts them to FLOAT and loads them directly. They are NULL when OWID did not compute them for a given country-day.

| Column | Table | What OWID computes |
|---|---|---|
| `new_cases_smoothed` | fact_covid_cases | 7-day rolling average of new_cases |
| `new_deaths_smoothed` | fact_covid_cases | 7-day rolling average of new_deaths |
| `new_cases_per_million` | fact_covid_cases | new_cases / population × 1,000,000 |
| `new_deaths_per_million` | fact_covid_cases | new_deaths / population × 1,000,000 |
| `daily_vaccinations_smoothed` | fact_vaccination | 7-day rolling average of daily vaccinations |
| `people_vaccinated_per_hundred` | fact_vaccination | people_vaccinated / population × 100 |
| `people_fully_vaccinated_per_hundred` | fact_vaccination | people_fully_vaccinated / population × 100 |
| `total_boosters_per_hundred` | fact_vaccination | total_boosters / population × 100 |
| `rolling_vaccinations_6m` | fact_vaccination | 6-month rolling vaccination total |
| `rolling_vaccinations_9m` | fact_vaccination | 9-month rolling vaccination total |
| `rolling_vaccinations_12m` | fact_vaccination | 12-month rolling vaccination total |
| `daily_occupancy_hosp_per_1m` | fact_hospitalization | occupancy per 1,000,000 population |
| `daily_occupancy_icu_per_1m` | fact_hospitalization | ICU occupancy per 1,000,000 population |
| `weekly_admissions_hosp_per_1m` | fact_hospitalization | weekly admissions per 1,000,000 population |
| `weekly_admissions_icu_per_1m` | fact_hospitalization | weekly ICU admissions per 1,000,000 population |

### 2. Computed at query time — analytical SQL, not ETL

These are **not stored** in the warehouse. They are derived by analytical queries in `sql/analytical_queries.sql` at report time, using the raw grain data already loaded.

| Metric | Query-time SQL pattern |
|---|---|
| 7-day rolling average (additional, not smoothed column) | `AVG(new_cases) OVER (PARTITION BY location_id ORDER BY date_id ROWS 6 PRECEDING)` |
| Day-over-day % change | `(today - yesterday) / NULLIF(yesterday, 0) * 100` using `LAG()` |
| 28-day cumulative cases | `SUM(new_cases) OVER (PARTITION BY location_id ORDER BY date_id ROWS 27 PRECEDING)` |
| Weekly aggregates | `SUM(new_cases) GROUP BY location_id, year, week_number` (via dim_date join) |
| Monthly aggregates | `SUM(new_cases) GROUP BY location_id, year, month` (via dim_date join) |
| Case fatality rate (CFR) | `total_deaths / NULLIF(total_cases, 0) * 100` |
